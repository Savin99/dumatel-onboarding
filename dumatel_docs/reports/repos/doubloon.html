    Doubloon -- Billing-сервис¶

    Назначение и роль в системе¶

    Standalone billing-микросервис платформы Dumatel. Инкапсулирует всю бизнес-логику платежей и подписок: управление балансом (hold/commit/release), тарифные планы и подписки, автопродление, привязка банковских карт и СБП, оплата через Tinkoff (T-Kassa), транзакции, учёт долга.

    Doubloon -- внутренний сервис, недоступный напрямую из интернета. Все запросы приходят от Sprut (API Gateway) через HTTP с авторизацией по X-Friend-Token. Является отдельным bounded context для биллинга с собственной БД (через общую библиотеку psdot).

    API / интерфейсы¶

    REST API (/api/billing -- 17 эндпоинтов)¶

    Endpoint Method Описание
    /api/billing/balance GET Баланс организации: quota, purchased, total, debt, tariff info, next_billing_date
    /api/billing/hold POST Резервирование кредитов (создание Freeze с operation_id)
    /api/billing/commit POST Списание зарезервированных кредитов (Freeze -> COMPLETED, ChargeOff)
    /api/billing/release POST Освобождение резерва (Freeze -> CANCELLED)
    /api/billing/buy POST Покупка дополнительных кредитов -> Tinkoff payment -> NATS anchor task
    /api/billing/change-plan POST Смена тарифного плана -> Tinkoff payment -> NATS anchor task
    /api/billing/tariffs GET Список активных тарифов с пометкой текущего
    /api/billing/transactions GET История транзакций организации
    /api/billing/transaction/{id}/complete POST Завершение транзакции (after payment confirmation)
    /api/billing/transaction/{id}/fail POST Пометка транзакции как failed
    /api/billing/process-auto-renewals/{id} POST Автопродление подписки -> Tinkoff -> NATS anchor task
    /api/billing/attach-card POST Инициация привязки карты -> Tinkoff -> NATS anchor_card task
    /api/billing/save-card POST Сохранение карты после подтверждения (pan, rebill_id, exp_date)
    /api/billing/card/make-primary PATCH Установка основной карты (карта или СБП)
    /api/billing/card/delete DELETE Удаление карты (+ Tinkoff removeCard) или СБП-аккаунта
    /api/billing/card/list GET Список карт и СБП-аккаунтов организации
    /api/billing/auto-renew POST Включение/выключение автопродления
    /api/health GET Healthcheck

    Биллинговая модель¶

    Типы баланса: - QUOTA -- квота от тарифного плана (обновляется ежемесячно) - PURCHASED -- дополнительно купленные кредиты (TTL 60 дней)

    Жизненный цикл кредитов: 

    Request -> hold(tokens) -> Freeze(ACTIVE, operation_id)
                |
     +----------+----------+
     |                      |
  commit(tokens)         release()
     |                      |
  Freeze(COMPLETED)    Freeze(CANCELLED)
  ChargeOff(amount)    (кредиты возвращены)

    Тарифы: - Free tier: квота в месяц (ограниченная) - Платные тарифы: расширенная квота, ежемесячная/годовая оплата - Дополнительные пакеты кредитов: покупка сверх квоты

    Подписки: - billing_day -- день биллинга - next_billing_date -- дата следующего списания - auto_renew -- флаг автопродления - Upgrade: немедленное применение, доплата разницы - Downgrade: запланированный на конец периода

    Платёжные методы: - Банковская карта (через Tinkoff T-Kassa, rebill для рекуррентных платежей) - СБП (Система Быстрых Платежей) -- добавлена недавно (MR qr_list)

    NATS Subjects (Producer)¶

    Subject Payload Потребитель
    anchor_tasks (configurable: settings.nats_anchor_subject) AnchorTask (transaction_id, payment_id, billing_period) Anchor (проверка статуса платежа в Tinkoff)
    anchor_card_tasks (configurable: settings.nats_anchor_card_subject) AnchorCardTask (payment_id, organization_id, transaction_id?, billing_period?) Anchor (проверка привязки карты)

    Аутентификация (service-to-service)¶

    Doubloon использует X-Friend-Token middleware для авторизации: - Все запросы (кроме /api/health) проверяют заголовок X-Friend-Token - Токен сравнивается с settings.service_auth_token - Без валидного токена -- HTTP 401 Unauthorized - JWT пользователя НЕ проверяется -- Doubloon доверяет вызывающему сервису (Sprut)

    Middleware-стек¶

    1. request_middleware -- логирование запросов (debug mode), X-Request-ID propagation к Sentry
    2. auth_middleware -- проверка X-Friend-Token (кроме /api/health)
    3. catch_exceptions_middleware -- централизованная обработка ошибок (DBError, RequestValidationError, HTTPException, generic Exception)

    Зависимости от других сервисов¶

    Сервис Протокол Направление Назначение
    PostgreSQL asyncpg (psdot) Doubloon -> PG Хранение: балансы, тарифы, подписки, транзакции, карты, freezes, charge-offs, долги, СБП-аккаунты
    NATS JetStream nats:// (orker) Doubloon -> NATS Публикация задач в Anchor: anchor_tasks, anchor_card_tasks
    Tinkoff T-Kassa HTTPS (aiohttp) Doubloon -> Tinkoff Платёжный процессинг: инициация платежа, рекуррентные списания, привязка/удаление карт
    Sentry HTTPS Doubloon -> Sentry Error tracking

    Вызывается из: | Вызывающий | Протокол | Сценарий | |---|---|---| | Sprut | HTTP (X-Friend-Token) | Все биллинговые операции из фронтенда (через /api/billing proxy) | | Bilge | HTTP (X-Friend-Token) | Автопродление подписок по расписанию (cron-like worker) | | Amigofish | HTTP (X-Friend-Token) | hold/commit/release кредитов при AI-генерации |

    Граф зависимостей¶

       Sprut ----+
   Bilge ----+----> Doubloon -----> PostgreSQL (asyncpg/psdot)
   Amigofish-+    (X-Friend-Token)     |
                       |           +--> NATS --> Anchor
                       |           |
                       +-----------+--> Tinkoff T-Kassa (HTTPS)

    Конфигурация (env-переменные)¶

    Ключевые настройки (без значений): - POSTGRES_HOST, POSTGRES_PORT, POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD - NATS_HOST, NATS_PORT - NATS_ANCHOR_SUBJECT (default: anchor_tasks) - NATS_ANCHOR_CARD_SUBJECT (default: anchor_card_tasks) - TINKOFF_API_URL (default: https://securepay.tinkoff.ru) - TINKOFF_TERMINAL_KEY, TINKOFF_PASSWORD - TINKOFF_TAXATION (default: osn), TAX (default: vat22) - BASE_AMOUNT (default: 1000) - SERVICE_AUTH_TOKEN -- для X-Friend-Token авторизации - SENTRY_DSN
